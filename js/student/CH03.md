# Processes and Process Control Blocks

**Lecture 2 — Processes and Process Control Blocks**

## Summary of Earlier Concepts

- A computer platform consists of a collection of hardware resources.
- Computer applications are developed to perform some task.
- It is inefficient for applications to be written directly for a given hardware platform.
- The OS was developed to provide a convenient, feature-rich, secure, and consistent interface for applications to use.
- The OS provides a uniform, abstract representation of resources that can be requested and accessed by applications.

## 1. What is a Process?

A process is:

- A sequence of instructions that executes.
- A program in execution — the entity that can be assigned to and executed on a processor.
- A unit of activity characterized by a single sequential thread of execution.
- An entity that can be traced by listing the sequence of instructions that execute.
- Associated with data needed by the program (context): all information the OS needs to manage the process, including a current state and an associated set of system resources.

### OS Management of Application Execution

- Resources are made available to multiple applications.
- The processor is switched among multiple applications so all will appear to be progressing.
- The processor and I/O devices can be used efficiently.

### Process Elements

Two essential elements of a process:

- Program code (text) — which may be shared with other processes executing the same program.
- A set of data associated with that code.

While the program is executing, a process can be uniquely characterized by elements including:

- Identifier
- State
- Priority
- Program counter
- Memory pointers
- Context data
- I/O status information
- Accounting information

### Process Control Block (PCB)

- The PCB contains the process elements and data needed by the OS to control the process.
- A PCB typically includes: identifier, state, priority, program counter, memory pointers, context data, I/O status information, and accounting information.
- It is possible to interrupt a running process and later resume execution as if the interruption had not occurred; the PCB makes this possible.
- The PCB is created and managed by the operating system and is a key tool that allows support for multiple processes.

### Trace and Dispatcher

- Trace: the behavior of an individual process by listing the sequence of instructions that execute (the processor behavior is the interleaving of traces of various processes).
- Dispatcher: a small program that switches the processor from one process to another.

## 2. Process States

### Two-State Process Model

- States: `Running` and `Not-running`.
- Dispatcher selects a process to run and moves processes between Running and Not-running states.
- Processes are often represented in a queue (Ready queue) where each entry is a pointer to a PCB.

### Origins of Process Creation (Table 3.1)

- New batch job: OS reads job control commands when prepared to take on new work.
- Interactive log-on: a user at a terminal logs on.
- Created by OS to provide a service (e.g., a process to control printing).
- Spawned by existing process: for modularity or parallelism.

### Process Spawning

- Parent process: original creating process.
- Child process: the new process.

### Process Termination

- There must be a means for a process to indicate completion (e.g., `exit()` or HALT instruction).
- For interactive applications, user actions (log off, quit) indicate completion.

Reasons for process termination (selected, from Table 3.2):

- Normal completion (process executes an OS service call to indicate completion).
- Time limit exceeded (various time measures: wall clock, CPU time, or inactivity for interactive processes).
- Memory unavailable (requires more memory than system can provide).
- Bounds violation (accessing memory not allowed).
- Protection error (attempting to use a resource improperly).
- Arithmetic error (e.g., division by zero).
- Time overrun (waited longer than allowed for an event).
- I/O failure (errors during I/O operations).
- Invalid instruction (attempts to execute nonexistent instruction).
- Privileged instruction (attempt to use an instruction reserved for OS).
- Data misuse (wrong type or uninitialized data).
- Operator or OS intervention (e.g., to break a deadlock).
- Parent termination (OS may terminate offspring when parent terminates).
- Parent request (parent may request termination of its children).

### Five-State Process Model (Figure 3.6)

States: `New` → `Ready` → `Running` → `Blocked` → `Exit` with transitions such as Admit, Dispatch, Release, Timeout, Event wait/occurs.

- `New`: created but not yet admitted to executable pool.
- `Ready`: process waiting to be dispatched.
- `Running`: process currently executing.
- `Blocked` (Waiting): cannot execute until an event occurs (I/O completion, semaphore, etc.).
- `Exit`: process released from pool of executable processes.

Transitions (common):

- Null → New: process created.
- New → Ready: OS admits the process (subject to memory limits).
- Ready → Running: selected by the OS.
- Running → Exit: process terminated.
- Running → Ready: end of time slice or preemption by higher-priority process.
- Running → Blocked: waiting for an event.
- Blocked → Ready: awaited event occurs.

### Queuing Models

- Ready queue: processes waiting to run.
- Blocked queue(s): processes waiting for specific events. Multiple blocked queues (one per event) avoid scanning a single blocked queue when an event occurs.

### Suspended Processes and Swapping

- Swapping: moving part or all of a process between main memory and disk.
- Suspend queue: a queue of processes temporarily removed from main memory (suspended).
- Reasons for suspension (Table 3.3): swapping to free memory, other OS reasons (e.g., suspecting a problem), interactive user request (debugging), timing (periodic execution), parent request.
- Suspended states: `Blocked/Suspend`, `Ready/Suspend` (two-suspend-state model allows swapping ready processes as well).

Characteristics of suspended processes:

- Not immediately available for execution.
- Placed in suspended state by an agent (itself, parent, or OS).
- May or may not be waiting on an event; may remain suspended until explicitly reactivated.

## 3. Process Description

What information does the OS need to control processes and manage resources for them? OS uses control tables and process descriptors.

### Control Tables (Figure 3.11)

- Memory tables: track main (real) and secondary (virtual) memory, allocation of memory to processes, protection attributes, and information to manage virtual memory.
- I/O tables: manage I/O devices and channels; track device availability, status of I/O operations, and memory locations used for transfers.
- File tables: provide information about existence, location on secondary memory, current status, and attributes of files (may be managed by a file management system or the OS).
- Process tables: primary process table and process images; must reference memory, I/O, and files.

### Process Control Structures

To manage a process the OS must know:

- Where the process is located (process image in memory or swapped out).
- Attributes necessary for management (state, identifiers, scheduling info, resource ownership, etc.).

Process image consists of:

- User data (modifiable part of user space, program data, user stack area).
- User program (code to be executed).
- Stack(s): LIFO stacks for parameters and return addresses for procedure and system calls.
- Process Control Block (PCB): data needed by the OS to control the process.

Table 3.4 (Typical elements of a process image): User Data, User Program, Stack, PCB.

### Typical Elements of a PCB (Table 3.5 — summarized)

1. Process Identification
   - Numeric identifiers: process ID, parent process ID, user ID.
2. Processor State Information
   - User-visible registers (e.g., general-purpose registers).
   - Control and status registers (program counter, condition codes, status flags).
   - Stack pointers.
   - Program status word (PSW) — contains condition codes and status (for x86, EFLAGS is an example).
3. Process Control Information
   - Scheduling and state information (process state, priority, scheduling-related information, event identity).
4. Data Structuring
   - Pointers to link processes in queues or parent-child relationships.
5. Interprocess Communication
   - Flags, signals, messages, and related information.
6. Process Privileges
   - Memory and instruction privileges, access to system utilities.
7. Memory Management
   - Pointers to segment/page tables describing virtual memory assigned to the process.
8. Resource Ownership and Utilization
   - Opened files, allocated devices, accounting/history of resource usage.

### Processor State and PSW

- Processor state: contents of processor registers (user-visible registers, control/status registers, stack pointers).
- PSW (Program Status Word): contains condition codes and status information (x86 `EFLAGS` is an example; flags include carry, parity, auxiliary carry, zero, sign, trap, interrupt enable, direction, overflow, IOPL, etc.).

### Role of the PCB

- The PCB is the most important data structure in an OS; contains all information needed by the OS about a process and is accessed by nearly every OS module (scheduling, interrupts, monitoring).
- Protection and controlled access to PCBs is important: bugs or design changes can have wide impact.

## 4. Process Control

### Modes of Execution

- User mode: less-privileged mode for user programs.
- Kernel (system/control) mode: more-privileged mode for the OS kernel; can execute privileged instructions and access protected memory regions.
- The processor typically indicates the current mode via a bit in the PSW.

### Typical Functions of an OS Kernel (Table 3.7)

- Process management: creation/termination, scheduling, dispatching, switching, synchronization, interprocess communication, management of PCBs.
- Memory management: allocation of address space, swapping, page/segment management.
- I/O management: buffer management, allocation of I/O channels and devices.
- Support functions: interrupt handling, accounting, monitoring.

### Process Creation Steps (summary)

When OS creates a new process:

1. Assign a unique process identifier.
2. Allocate space for the process (program, data, stacks, PCB).
3. Initialize the PCB.
4. Set appropriate linkages (lists, queues).
5. Create or expand other data structures (e.g., accounting records).

### Mechanisms for Interrupting Execution (Table 3.8)

- Interrupt: asynchronous external event (clock interrupt, I/O interrupt, memory fault).
- Trap: an error or exception generated within the currently running process (used to handle faults or exceptions).
- Supervisor call (system call): explicit request by a process to call an OS service.

System interrupt handling:

- On an interrupt, the processor sets the program counter to an interrupt handler and switches to kernel mode so privileged instructions can be used.
- Time slice: maximum amount of time a process may execute before being interrupted.

### Mode Switching vs Process Switching

- Mode switch: switch between user and kernel mode (lower overhead) when handling an interrupt/trap/syscall within the same process.
- Full process switch: involves saving/restoring processor context and updating PCBs and memory structures.

Change of process state (full process switch — typical steps):

1. Save the context of the processor (PC and registers).
2. Update the PCB of the currently running process.
3. Move the PCB to the appropriate queue (Ready, Blocked, etc.).
4. Select another process for execution.
5. Update the PCB of the selected process.
6. Update memory-management data structures (if required).
7. Restore the context of the selected process and resume execution.

## UNIX / Linux Notes (selected material)

### Process Creation in UNIX

- `fork()` system call creates a new process (child) as a copy of the parent; `exec()` can be used after `fork()` to replace the process memory space with a new program.
- `fork()` actions (typical): allocate slot in process table, assign unique PID, copy process image (except shared memory), increment file ownership counters, place child in Ready state, return child's PID to parent and 0 to child.
- After creation, the kernel may resume parent, transfer control to child, or dispatch another process.

### Process Identifiers and Trees

- Processes are managed via PIDs; parent/child relationships form process trees (e.g., `init` pid 1 → `login` → `bash`, etc.).

### Resource and Execution Options

- Resource sharing: parent and children may share all, some, or no resources.
- Execution: parent and children may execute concurrently or parent may wait for children to terminate.
- Address space: child may be a duplicate of parent or have a new program loaded.

### Process Representation in Linux

- In Linux, a process is represented by a `task_struct` C structure (fields include PID, state, scheduling info, parent pointer, children list, files, memory descriptor `mm_struct`, etc.).
- All active processes are represented using a doubly linked list of `task_struct`; the kernel maintains a pointer `current` to the process currently executing.

### UNIX Process Table Entry (selected fields)

- Process status (current state), pointers to U area and memory, process size, user identifiers (real/effective UID), process identifiers (PID and parent PID), event descriptor (sleeping state), priority, signal pending, timers (CPU time, resource usage), queue pointers, and memory status (in-memory or swapped out).

## Examples, Illustrations, and Practical Points

- Dispatcher: small program switching processor among processes.
- Traces and combined traces illustrate interleaving of processes and how dispatching creates apparent concurrency.
- Multiprocess architectures (example: Chrome) use separate processes for browser UI, renderers (per tab/site), and plug-ins for isolation and crash containment.

---

## Role and Importance of PCBs

- PCBs are central to OS process management and are accessed by scheduling, interrupt handling, and monitoring modules.
- Careful design and protection of PCBs are necessary to avoid system corruption and ensure modularity.
